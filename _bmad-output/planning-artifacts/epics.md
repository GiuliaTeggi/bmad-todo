# todo-app - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for todo-app, decomposing the requirements from the PRD and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Users can view a list of all todos
FR2: Users can see a clear visual distinction between active and completed todos
FR3: Users can create a new todo with a text description
FR4: Users can toggle the completion status of an individual todo
FR5: Users can delete an individual todo
FR6: Users can see a meaningful empty state when no todos exist
FR7: The system persists all todos across page refreshes
FR8: The system persists all todos across browser sessions
FR9: All todo state is stored server-side and treated as authoritative
FR10: The UI reflects a newly created todo immediately, without waiting for server confirmation
FR11: The UI reflects a toggled completion status immediately, without waiting for server confirmation
FR12: The UI reflects a deleted todo immediately, without waiting for server confirmation
FR13: The UI reverts an optimistic change if the corresponding API call fails
FR14: Users see a loading indicator while todos are being fetched on initial load
FR15: Users see visual confirmation (animation/strikethrough) when a todo is marked complete
FR16: Users see a user-facing error message when an API call fails
FR17: All interactive elements are keyboard accessible
FR18: All interactive elements are screen-reader compatible with appropriate ARIA labels
FR19: Focus is managed correctly after dynamic content changes (create, delete)
FR20: The UI is fully functional on screen widths from 375px upward
FR21: The UI is fully functional on touch-based mobile devices
FR22: The backend exposes RESTful API endpoints for all todo operations (list, create, complete, delete)
FR23: The backend validates all incoming request data against a defined schema
FR24: The backend returns appropriate HTTP error responses with descriptive error information
FR25: The full application stack starts with a single `docker-compose up` command with no manual setup

### NonFunctional Requirements

NFR1: Initial page load completes within 2 seconds on a standard broadband connection
NFR2: All todo create/complete/delete operations update the UI within one animation frame via optimistic updates — perceived latency is effectively zero
NFR3: API responses for all CRUD operations complete within 500ms under normal conditions
NFR4: No critical OWASP Top 10 vulnerabilities (injection, XSS, broken access control, etc.)
NFR5: All API inputs are validated and sanitised server-side before processing
NFR6: The application does not expose internal error details in API responses (stack traces, database errors)
NFR7: Dependencies are sourced from trusted registries and kept reasonably up to date
NFR8: WCAG 2.1 AA compliance — zero critical violations as reported by an automated audit tool (e.g., axe)
NFR9: All interactive elements are operable via keyboard alone
NFR10: All meaningful UI elements have accessible names (via labels or ARIA)
NFR11: Colour contrast ratios meet WCAG AA minimums for all text and interactive elements
NFR12: All todo mutations (create, complete, delete) are either confirmed persisted or the user is notified of failure — no silent data loss
NFR13: The application handles API failures gracefully without crashing or entering an unrecoverable state
NFR14: Data is consistent between the frontend and backend after any operation completes or fails

### Additional Requirements

- **Starter Template (Architecture — impacts Epic 1 Story 1):** pnpm workspaces monorepo; project must be initialized before any feature implementation. Commands: `pnpm create vite@latest apps/frontend -- --template react-ts`, manual Fastify backend scaffold, and `packages/shared` setup.
- **Monorepo structure:** pnpm workspaces with `apps/frontend`, `apps/backend`, `packages/shared`; shared types exported as `@bmad-todo/shared`
- **Database:** PostgreSQL (latest stable) via Drizzle ORM; migrations generated by Drizzle Kit and run automatically on container startup via an init script
- **Drizzle schema:** `apps/backend/src/db/schema.ts` — single source of truth for DB shape; `snake_case` column names, nullable `user_id` placeholder for future auth
- **ORM:** Drizzle ORM with full TypeScript type inference; Drizzle Kit for migration management
- **State management:** TanStack Query v5 for all server state; the mandatory optimistic update pattern (`onMutate` / `onError` / `onSettled`) must be applied to all three mutations (create, toggle, delete)
- **No client-side router** — single-screen SPA; no navigation
- **Styling:** Plain CSS with CSS custom properties; mobile-first; touch targets ≥44×44px (WCAG 2.5.5)
- **Security:** `@fastify/helmet` for HTTP security headers; CSP restricting script sources; CORS restricted to `FRONTEND_ORIGIN` env var in production
- **Accessibility:** ARIA live regions for dynamic list changes; explicit focus management after create (input cleared and refocused) and delete (focus moved to next item or input)
- **Testing infrastructure:** Vitest for frontend unit/integration tests; `node:test` (Node.js built-in) for backend unit/integration tests; Playwright for E2E (≥5 scenarios); ≥70% meaningful test coverage overall
- **Docker Compose:** 3 services — `frontend` (nginx, port 80), `backend` (Node.js Fastify, port 3000), `db` (PostgreSQL, port 5432 internal); named volume for PostgreSQL data persistence; multi-stage Dockerfiles for both frontend and backend
- **Environment variables:** `DATABASE_URL`, `FRONTEND_ORIGIN`, `NODE_ENV` — no secrets hardcoded; `.env.example` documents all required vars
- **Local development:** `pnpm dev` at workspace root starts both Vite and tsx watch concurrently
- **Naming conventions:** `snake_case` DB columns; `camelCase` JSON fields and TS identifiers; `PascalCase` React components; `useCamelCase` hooks; `camelCaseSchema` Zod schemas
- **Error codes:** `TODO_NOT_FOUND`, `VALIDATION_ERROR`, `INTERNAL_ERROR`
- **API response format:** No envelope wrapper; success returns resource or array directly; `DELETE` returns `204 No Content`; dates as ISO 8601 strings

## Epic 1: Working Todo Backend

A fully functional REST API with PostgreSQL persistence. A developer can `curl` any endpoint and perform all CRUD operations. The monorepo scaffold and shared types package are in place, enabling the frontend to be built next.

### Story 1.1: Initialize Monorepo Workspace

As a developer,
I want a working pnpm workspaces monorepo with a shared TypeScript types package,
So that I can build the frontend and backend with shared, type-safe API contracts from the start.

**Acceptance Criteria:**

**Given** a clean checkout
**When** I run `pnpm install`
**Then** `apps/frontend`, `apps/backend`, and `packages/shared` workspaces all resolve correctly
**And** `packages/shared` is importable as `@bmad-todo/shared` from both apps

**Given** the workspace setup
**When** I inspect TypeScript configuration
**Then** strict mode is enabled in all packages
**And** `packages/shared/src/index.ts` exports `Todo`, `CreateTodoRequest`, and `UpdateTodoRequest` interfaces

**Given** the root `package.json`
**When** inspected
**Then** it contains no direct dependencies (workspace root has scripts only)
**And** `pnpm-workspace.yaml` declares `apps/*` and `packages/*`

**Given** a clean checkout
**When** I run `pnpm dev` at the workspace root
**Then** the Vite dev server starts on port 5173 and the backend starts on port 3000 concurrently

### Story 1.2: Todo Database Schema and Migrations

As a developer,
I want a PostgreSQL `todos` table managed by Drizzle ORM with automated migrations,
So that the application can persist todo data reliably with full type safety.

**Acceptance Criteria:**

**Given** a running PostgreSQL instance (via `DATABASE_URL` env var)
**When** the backend initialises
**Then** the Drizzle migration runs automatically and creates the `todos` table

**Given** the `todos` table
**When** the schema is inspected
**Then** it has columns: `id` (uuid, primary key), `text` (varchar, not null), `is_completed` (boolean, default false), `created_at` (timestamp with timezone, default now()), `user_id` (uuid, nullable — future auth placeholder)
**And** all column names are `snake_case`

**Given** the Drizzle query layer
**When** results are returned from a select query
**Then** fields are fully typed in TypeScript with `camelCase` names (`isCompleted`, `createdAt`)

**Given** the migration files
**When** committed to the repository
**Then** a SQL file exists at `apps/backend/src/db/migrations/0000_init.sql`

### Story 1.3: Todo REST API Endpoints

As a developer,
I want all four CRUD REST API endpoints for todos with Zod schema validation and standard error responses,
So that any client can manage todos over HTTP with guaranteed input safety.

**Acceptance Criteria:**

**Given** the backend is running
**When** `GET /api/todos` is called
**Then** the response is `200 OK` with an array `[{ id, text, isCompleted, createdAt }]`
**And** an empty database returns `[]`

**Given** the backend is running
**When** `POST /api/todos` is called with `{ "text": "Buy milk" }`
**Then** the response is `201 Created` with `{ id, text, isCompleted: false, createdAt }`
**And** the record is persisted in the database

**Given** `POST /api/todos` is called with an empty `text` or missing body
**When** the request is processed
**Then** the response is `400 Bad Request` with `{ "error": { "message": "...", "code": "VALIDATION_ERROR" } }`
**And** no record is created

**Given** a todo with a known `id`
**When** `PATCH /api/todos/:id` is called
**Then** the response is `200 OK` with the updated todo (`isCompleted` toggled)

**Given** a todo with a known `id`
**When** `DELETE /api/todos/:id` is called
**Then** the response is `204 No Content`
**And** the record no longer exists in the database

**Given** a non-existent `id`
**When** `PATCH` or `DELETE` is called
**Then** the response is `404 Not Found` with `{ "error": { "message": "...", "code": "TODO_NOT_FOUND" } }`

**Given** any API error
**When** the response is returned
**Then** it never contains a stack trace, SQL error message, or internal system detail

**Given** the Fastify server starts
**When** any response is returned
**Then** `@fastify/helmet` security headers are present
**And** CORS allows only the origin specified in the `FRONTEND_ORIGIN` environment variable

### Story 1.4: Backend Unit and Integration Test Suite

As a developer,
I want a `node:test` test suite for all API routes and the database layer,
So that backend behaviour is verified and regressions are caught without external test dependencies.

**Acceptance Criteria:**

**Given** the backend test suite
**When** `pnpm test` runs in `apps/backend`
**Then** all tests pass
**And** coverage is ≥70% of backend source files

**Given** `routes/todos.test.ts` and `routes/todo.test.ts`
**When** they run
**Then** every endpoint (GET, POST, PATCH, DELETE) is tested for: happy path, 400 validation error, and 404 not-found error

**Given** the test setup
**When** tests run
**Then** each test uses isolated database state (setup/teardown between tests)
**And** the Fastify app is imported from `app.ts` (not `server.ts`, keeping `listen()` out of tests)

As a user,

**Acceptance Criteria:**

As a user,

### Story 3.1: Docker Compose Infrastructure

**Given** the `.env.example` file

### Story 3.2: Playwright End-to-End Test Suite

As a developer,
I want ≥5 Playwright E2E tests that verify the full application works end-to-end against the running stack,
So that critical user journeys are regression-tested at the integration level.

**Dev Note:** The implementing agent should have the Playwright MCP server configured in `.vscode/mcp.json` to interactively inspect the running app and validate test scenarios before committing them:

```json
{
  "servers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

**Acceptance Criteria:**

**Given** the full stack running (via Docker Compose or local `pnpm dev`)
**When** `pnpm test:e2e` runs
**Then** all Playwright tests pass

**Given** the E2E suite
**When** inspected
**Then** it contains at minimum these 5 scenarios:

1. **Create a todo** — type text, submit, verify it appears in the list
2. **Toggle a todo complete** — click checkbox, verify strikethrough appears

4. **Delete a todo** — click delete button, verify item is removed from the list

**Given** an E2E test runs

**Then** it uses accessible queries (by role/label) rather than CSS selectors or test IDs wherever possible

**Given** the Playwright configuration (`playwright.config.ts`)
**When** inspected
**Then** it targets `http://localhost` (production Docker stack) or a configurable base URL
**And** E2E test files live in `apps/frontend/e2e/todos.spec.ts`
